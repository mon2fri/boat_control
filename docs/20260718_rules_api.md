# Validation Rules Configuration API

## Endpoints

### `GET /api/rules/`

Returns all rules from the configuration file.

**Response (200):**
```json
{
  "version": 1,
  "rules": [
    {
      "rule_id": "R001",
      "name": "Status Check",
      "description": "Validates status field",
      "conditions": [
        {"column_name": "type", "operator": "eq", "filter_value": "premium"}
      ],
      "condition_relation": "and",
      "logic": {
        "format": "value_vs_column",
        "column_name": "status",
        "operator": "eq",
        "target_value": "active"
      }
    }
  ]
}
```

### `POST /api/rules/`

Creates a new rule with auto-incrementing ID.

**Request:**
```json
{
  "name": "Status Check",
  "description": "Validates status field",
  "conditions": [
    {"column_name": "type", "operator": "eq", "filter_value": "premium"}
  ],
  "condition_relation": "and",
  "logic": {
    "format": "value_vs_column",
    "column_name": "status",
    "operator": "eq",
    "target_value": "active"
  }
}
```

**Response (201):**
```json
{"rule_id": "R001", "message": "Rule created."}
```

### `GET /api/rules/<rule_id>/`

Returns a specific rule by ID.

### `PUT /api/rules/<rule_id>/`

Updates an existing rule.

### `DELETE /api/rules/<rule_id>/`

Deletes a rule.

### `POST /api/rules/remote/`

Loads rules from a remote YAML file URL.

**Request:**
```json
{"url": "https://example.com/rules.yaml"}
```

## Rule Schema

### Rule Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Human-readable rule name |
| `description` | string | No | Optional description |
| `conditions` | array | No | List of condition objects |
| `condition_relation` | string | Conditional | `"and"` or `"or"` (required if 2+ conditions) |
| `grouping` | array | No | Grouping for 3+ conditions |
| `logic` | object | Yes | Logic clause (see below) |

### Condition Object

| Field | Type | Values |
|-------|------|--------|
| `column_name` | string | Column name to filter on |
| `operator` | string | `eq`, `neq`, `contains`, `ncontains` |
| `filter_value` | string | Value to compare against |

### Logic Clause

| Field | Type | Values |
|-------|------|--------|
| `format` | string | `value_vs_column`, `column_vs_column` |
| `column_name` | string | Left-hand column |
| `operator` | string | Comparison operator |
| `target_value` | string | Right-hand value or column name |

## Rule IDs

- Auto-incrementing format: `R001`, `R002`, etc.
- IDs are never reused after deletion
- Starting index preserved in YAML file

## Persistence

- Rules stored in `config/rules/rules.yaml`
- Atomic writes via temporary file + rename
- Version field tracks schema changes
- `next_index` tracks next available ID
